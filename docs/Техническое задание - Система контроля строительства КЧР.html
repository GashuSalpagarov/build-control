Система контроля строительных объектов для Министерства строительства КЧР

ЭТАП 1: РОЛИ И ПРАВА ДОСТУПА

1.1 Суперадмин (разработчики - мы)

Функции:

  • Создание республик в системе (КЧР, Адыгея, Дагестан и т.д.)
  • Создание учётных записей министров для каждой республики
  • Имперсонация - возможность зайти в учётку любого министра и работать от его имени (для техподдержки)
  • Полный доступ ко всем данным всех республик
  • Управление подписками и тарифами

Доступ:

  • Видит ВСЁ по всем республикам
  • Может изменять любые данные
  • Может логиниться как любой пользователь

1.2 Правительство (Губернатор и его аппарат)

Функции:

  • Только просмотр (read-only) - никаких изменений
  • Видят ВСЁ по всем объектам своей республики
  • Могут выгружать отчёты
  • Могут смотреть дашборды и аналитику

Доступ:

  • Видят все объекты республики
  • Видят все финансы
  • Видят все планы-графики
  • Видят всех пользователей

Ограничения:

  • ❌ Не могут ничего изменять
  • ❌ Не могут создавать объекты/пользователей
  • ❌ Не могут отвечать на обращения
  • Привязаны только к своей республике

1.3 Министр (главный администратор республики)

Функции:

  • ✅ Создание объектов строительства (название, адрес, сроки, бюджет)
  • ✅ Создание и управление подрядчиками (ООО, ИП - ИНН, название, контакты)
  • ✅ Создание подведомственных организаций (технадзор)
  • ✅ Создание пользователей: проверяющих, бухгалтеров, технадзор
  • ✅ Привязка пользователей к объектам (один пользователь может быть на нескольких объектах)
  • ✅ Внесение и редактирование плана-графика работ (после утверждения на бумаге)
  • ✅ Утверждение плана-графика в системе
  • ✅ Просмотр всех обращений подрядчиков
  • ✅ Ответы на обращения подрядчиков
  • ✅ Полный доступ ко всем данным своей республики
  • ✅ Просмотр дашборда по всем объектам

Доступ:

  • Видит все объекты республики
  • Видит всех пользователей
  • Видит всю финансовую информацию
  • Видит все обращения
  • Видит всю аналитику

Ограничения:

  • Привязан только к своей республике
  • Не может создавать другие республики
  • Не может видеть данные других республик

1.4 Бухгалтер

Функции:

  • ✅ Внесение данных о платежах подрядчикам через казначейство
  • ✅ Указание данных платежа:
    • Дата платежа
    • Сумма в рублях
    • За что платёж (этап работ или описание типа "аванс 30%")
    • Номер платёжного поручения
    • Комментарий (опционально)

Доступ:

  • Видит только объекты, к которым привязан
  • Видит финансовую информацию только по своим объектам
  • Видит название этапов работ (чтобы выбрать за что платёж)

Ограничения:

  • ❌ НЕ видит план-график работ (детали)
  • ❌ НЕ видит фактические данные (люди, техника)
  • ❌ НЕ может редактировать объекты
  • ❌ НЕ может видеть другие объекты
  • ❌ НЕ видит обращения подрядчиков

1.5 Технадзор (Подведомственная организация)

Функции:

  • Получение бумажного плана-графика от подрядчика
  • Участие в утверждении плана на бумаге совместно с министром
  • Внесение утверждённого плана-графика в систему через веб-форму
  • Редактирование плана-графика (после утверждения изменений на бумаге министром)
  • Еженедельная проверка объёмов выполненных работ:
    • Запись физических объёмов (например: "выкопано 30 м³", "уложено 45 м³ бетона")
    • Процент выполнения работы
    • Комментарии к выполнению
  • Просмотр обращений подрядчиков по объектам
  • Ответы на обращения подрядчиков (первичные)

Доступ:

  • Видит ВСЕ объекты республики
  • Видит планы-графики всех объектов
  • Видит обращения подрядчиков
  • Видит фактические данные (люди, техника) по всем объектам

Ограничения:

  • Не может создавать объекты
  • Не может создавать пользователей
  • НЕ видит финансовую информацию (платежи)
  • Не может утверждать план (только министр)
Примечание по проверке объёмов:
  • Формат записи: по каждой работе указывается физический объём
  • Примеры: "выкопано 30 м³", "уложено 45 м³ бетона", "смонтировано 120 м² кровли"
  • Можно вводить процент выполнения вручную или система считает автоматически
  • Обязательно указывать дату проверки и ФИО проверяющего

1.6 Проверяющий (ежедневный контроль ресурсов на объекте)

Функции:

  • Ежедневная фиксация посещения объекта
  • Запись количества рабочих на объекте в этот день
  • Запись количества и видов техники:
    • Вид техники (например: Камаз, Экскаватор, Кран)
    • Количество единиц
  • Добавление комментариев (опционально)
  • Просмотр истории своих проверок

Доступ:

  • Видит только объекты, к которым привязан
  • Видит только интерфейс для ввода данных
  • НЕ видит плановые показатели (сколько должно быть людей/техники) - чтобы не подгонял данные под план

Ограничения:

  • Не видит финансы
  • Не видит план-график работ
  • Не видит проверки объёмов (технадзор)
  • Не видит обращения подрядчиков
  • Только ввод фактических данных по своим объектам
Особенность: Проверяющий видит ТОЛЬКО форму ввода, без графиков и аналитики. Это важно для объективности данных.

1.7 Подрядчик

Функции:

  • Просмотр своих объектов
  • Просмотр плана-графика своих объектов
  • Просмотр фактического выполнения (успевает или отстаёт)
  • Создание обращений/вопросов к министру и технадзору:
    • Привязка к конкретному объекту и этапу работ
    • Описание проблемы (например: "выкопали, наткнулись на трубу, работа остановлена")
    • Прикрепление фото (опционально)
  • Просмотр ответов на свои обращения
  • Просмотр истории переписки

Обращения подрядчика:

  • Статусы: Новое → На рассмотрении → Решено
  • Первично отвечает технадзор, но министр тоже видит и может ответить
  • История переписки сохраняется
  • Уведомления о новых ответах

Доступ:

  • Видит только объекты, где он является подрядчиком
  • Видит план-график только своих объектов
  • Видит фактическое выполнение (графики, проценты)
  • Видит свои обращения и ответы

Ограничения:

  • Не может редактировать план
  • НЕ видит финансы (платежи)
  • Не может видеть объекты других подрядчиков
  • Не видит данные о ежедневных проверках (люди, техника)
  • Не может создавать/редактировать пользователей

ЭТАП 2: ВНЕСЕНИЕ ПЛАНА-ГРАФИКА В СИСТЕМУ

2.1 Общая логика

ВАЖНО:
  • План-график утверждается НА БУМАГЕ (подписи, печати) ДО внесения в систему
  • Утверждение происходит вне системы (совещание, министр + технадзор + подрядчик)
  • Система НЕ содержит workflow утверждения
  • В систему вносится уже утверждённый план
  • После внесения план сразу становится активным и начинается контроль

2.2 Кто вносит план

Варианты:

  1. Технадзор - вносит план через веб-форму (основной способ)
  2. Министр - может внести план сам (резервный вариант)

НЕ вносят:

  • Проверяющий - не имеет доступа к планам
  • Бухгалтер - не видит план-график
  • Подрядчик - только читает свой план

2.4 ВАЖНЫЕ ОСОБЕННОСТИ

2.4.1 Ресурсы "в день"

КРИТИЧЕСКИ ВАЖНО:

  • Все количества ресурсов (люди, техника) указываются НА день
  • Это значит: "8 человек" = "каждый день на объекте должно быть 8 человек"
  • "2 бетономешалки" = "каждый день должно быть 2 бетономешалки"
  • Даже если в бумажном плане написано просто "8 человек, 2 экскаватора" - это ВСЕГДА означает "в день"

Примеры:

  • План: "Подготовительные работы, 01.01-10.01, 8 человек, 1 Камаз"
  • Значение: С 1 по 10 января КАЖДЫЙ ДЕНЬ должно быть 8 человек и 1 Камаз

2.4.2 Проверка корректности дат

Автоматические валидации:

  1. Дата окончания проекта ≥ дата начала проекта
  2. Дата начала этапа ≥ дата начала проекта
  3. Дата окончания этапа ≤ дата окончания проекта
  4. Дата окончания этапа ≥ дата начала этапа
  5. Этапы МОГУТ пересекаться по датам - это нормально для строительства (например: этап "Вырыть котлован" 01.01-10.01 и этап "Очистка территории" 04.01-08.01 могут идти параллельно)

Показать предупреждения:

  • Если сумма бюджетов этапов > общего бюджета
  • Если сумма бюджетов этапов < общего бюджета (возможно, забыли этап)

2.4.3 Справочник техники

В системе должен быть заранее созданный справочник видов техники:

  • Камаз
  • Экскаватор
  • Бетономешалка
  • Автокран
  • Погрузчик
  • Бульдозер
  • Виброплита
  • Компрессор
  • Автобетоносмеситель
  • Автовышка
  • Самосвал
  • Грузовик
  • [+ Добавить свой вид] - можно добавить произвольное название

Возможность расширения:

  • Суперадмин и Министр могут добавлять новые виды техники в справочник
  • При вводе нового вида техника автоматически добавляется в общий справочник

2.6 Что происходит после внесения плана

После нажатия [Завершить]:

  1. План сохраняется в базу данных
  2. План автоматически получает статус "Активный" (не "На утверждении"!)
  3. Объект появляется в общем списке объектов
  4. Подрядчик автоматически получает доступ к своему объекту
  5. Система начинает отсчитывать дни и показывать текущий статус
  6. Проверяющий может начинать ежедневные проверки ресурсов
  7. Технадзор может начинать еженедельные проверки объёмов
  8. Бухгалтер видит объект и может вносить платежи

Уведомления (опционально):

  • Подрядчику на email: "Объект добавлен, вы можете войти в систему"
  • Проверяющим: "Новый объект для контроля"

2.7 Редактирование плана

Когда может понадобиться:

  • Ошибка при вводе данных
  • Изменение сроков (согласованное с министром)
  • Добавление/удаление этапов

Кто может редактировать:

  • Министр - может редактировать всё
  • Технадзор - может редактировать план-график (только после утверждения изменений министром на бумаге)
  • Остальные - не могут

Логика редактирования:

  1. Открыть объект
  2. Кнопка [Редактировать план]
  3. Открывается тот же мастер, но с уже заполненными данными
  4. Можно изменить любые поля
  5. Можно добавить новые этапы
  6. Можно удалить этапы (если по ним ещё нет фактических данных)
  7. [Сохранить изменения]
ВАЖНО:
  • Если по этапу уже есть проверки (объёмы, ресурсы) - нельзя удалить этот этап
  • Можно изменить название, даты, бюджет - но нельзя удалить

ЭТАП 3: ЕЖЕДНЕВНЫЙ КОНТРОЛЬ РЕСУРСОВ (ПРОВЕРЯЮЩИЙ)

3.1 Описание роли "Проверяющий"

Кто это:

  • Человек, который физически приходит на строительный объект каждый рабочий день
  • Считает количество рабочих на объекте
  • Фиксирует присутствующую технику и её количество
  • Отправляет данные в систему через простую форму

Это НЕ:

  • НЕ технадзор (технадзор проверяет объёмы работ еженедельно)
  • НЕ инспектор качества
  • НЕ принимает решений и не оценивает качество

Суть роли:

  • Простой "счетчик" людей и техники
  • Фиксирует ФАКТ присутствия ресурсов на объекте
  • Объективный наблюдатель

Кем может быть:

  • Сотрудник министерства
  • Представитель подведомственной организации
  • Специально нанятый человек для ежедневного мониторинга
Важная особенность:
  • Может быть несколько проверяющих на одном объекте (работают посменно или дублируют друг друга)
  • Каждый проверяющий привязывается к нескольким объектам (маршрут объезда)

3.3 Форма ежедневной проверки

Поле Тип Обязательность Описание

3.4 Логика работы формы

3.4.1 Список техники

Откуда берется список техники:

  • Система подтягивает виды техники из плана-графика этого объекта (на текущий период)
  • Показывает проверяющему список техники, которую нужно отметить
  • ВАЖНО: Проверяющий видит НАЗВАНИЯ техники (Камаз, Экскаватор), но НЕ видит плановое КОЛИЧЕСТВО
  • Изначально все поля количества пустые (или = 0)
  • Проверяющий вводит фактическое количество каждого вида техники
Пример:
По плану на сегодня должно быть: 2 Камаза, 1 Экскаватор, 2 Бетономешалки

Проверяющий видит форму:
□ Камаз: [___] шт.
□ Экскаватор: [___] шт.
□ Бетономешалка: [___] шт.

НЕ видит: что должно быть 2, 1, 2
Важно:
  • Проверяющий может отмечать ТОЛЬКО ту технику, которая указана в плане на текущий период
  • НЕЛЬЗЯ добавить технику, которой нет в списке
  • Если на объекте есть техника не из плана - проверяющий пишет об этом в комментарии
  • Если запланированной техники нет на объекте - ставим "0"

3.4.2 Количество рабочих

Обязательное поле:

  • Нельзя отправить форму без указания количества рабочих
  • Можно указать "0" (если объект пустой - выходной, дождь и т.д.)
  • При "0" желательно написать комментарий (но не обязательно)

3.4.3 Комментарий

Необязательное поле:

  • Проверяющий сам решает, нужен ли комментарий
  • Рекомендуется писать при:
    • Погодных условиях (дождь, снег, мороз)
    • Отсутствии людей/техники
    • Любых необычных ситуациях

Примеры комментариев:

  • "Сильный дождь, работы не ведутся"
  • "Экскаватор на ремонте, ожидается завтра"
  • "Выходной день"
  • "Работа идет в обычном режиме"
  • Пустое поле (если все стандартно)

3.5 Редактирование сегодняшней проверки

Логика:

  • Если проверяющий уже отметился сегодня, он может отредактировать свою проверку
  • Кнопка [Редактировать] открывает ту же форму, но с уже заполненными данными
  • Можно изменить любые поля (количество людей, техника, комментарий)
  • После сохранения показывается уведомление: "Проверка обновлена"

Ограничение:

  • Редактировать можно только сегодняшнюю проверку
  • Проверки прошлых дней редактировать нельзя (только просмотр в истории)

Причина:

  • Вдруг проверяющий не заметил человека или единицу техники
  • Вдруг ошибся при вводе цифры
  • Вдруг забыл написать важный комментарий

НЕ видит проверки других проверяющих:

НЕ видит аналитику и дашборды:

  • Избежать подгонки факта под план

  • Сфокусировать проверяющего только на фиксации фактов

ЭТАП 4: ВНЕСЕНИЕ ПЛАТЕЖЕЙ (БУХГАЛТЕР)

4.1 Описание роли "Бухгалтер"

Кто это:

  • Сотрудник бухгалтерии министерства
  • Отвечает за учёт платежей подрядчикам через казначейство
  • Фиксирует факт перечисления денег

Основная функция:

  • Внесение информации о платежах подрядчикам в систему
  • Привязка платежа к конкретному этапу работ
  • Контроль сколько уже оплачено по каждому этапу

Это НЕ:

  • НЕ принимает решения о платежах (решения принимаются вне системы)
  • НЕ утверждает платежи (утверждение через казначейство)
  • НЕ оценивает качество работ
  • НЕ контролирует ход строительства

Суть роли:

  • Простая фиксация платежей в системе
  • Учёт финансовых операций по объектам
  • Прозрачность расходования бюджета
Важная особенность:
  • Бухгалтер привязан к нескольким объектам (ведёт несколько строек)
  • Видит только финансовую информацию, не видит план-график и фактическое выполнение
  • НЕ может редактировать платежи после добавления (только создавать новые)

4.4 Форма добавления платежа

Поле Тип Обязательность Описание
Этап работ Выбор из списка Обязательно Выбор этапа из плана-графика
Дата платежа Дата Обязательно Дата перечисления средств
Сумма платежа Число (рубли) Обязательно Сумма в рублях
Номер п/п Текст Необязательно Номер платёжного поручения
Комментарий Текст Необязательно Пояснение к платежу

4.5 Логика работы формы

4.5.1 Выбор этапа работ

Откуда берется список этапов:

  • Система показывает все этапы из плана-графика этого объекта
  • Этапы отображаются в хронологическом порядке
  • Для каждого этапа показан бюджет

После выбора этапа автоматически появляется информационный блок:

  • Бюджет этапа
  • Уже оплачено (сумма и процент)
  • Осталось оплатить
Пример:
Этап: Фундаментные работы
Бюджет: 1 800 000 ₽
Оплачено: 900 000 ₽ (50%)
Осталось: 900 000 ₽

Важно:

  • Можно оплатить больше 100% бюджета этапа (дополнительные работы)
  • Можно оплатить меньше 100% (частичная оплата)
  • Система НЕ блокирует переплату, только предупреждает

4.5.2 Дата платежа

Обязательное поле:

  • Дата фактического перечисления денег
  • Можно выбрать любую дату (прошлую, текущую, будущую)
  • По умолчанию = сегодняшняя дата

Примечание:

  • Если платёж был вчера, а бухгалтер вносит сегодня - выбирает вчерашнюю дату

4.5.3 Сумма платежа

Обязательное поле:

  • Сумма в рублях
  • Целое число (без копеек)
  • Минимум: 1 рубль
  • Максимум: не ограничен
Предупреждение при переплате:

Если сумма платежа больше остатка по этапу:

Внимание!
Сумма платежа (1 500 000 ₽) больше остатка по этапу (900 000 ₽).
Вы уверены, что хотите продолжить?
[Да, продолжить] [Отмена]

4.5.4 Номер платёжного поручения

Необязательное поле:

  • Свободный текст
  • Обычно вводят: "№12345 от 20.01.2025"
  • Можно оставить пустым

Зачем:

  • Для связи с бухгалтерскими документами
  • Для аудита и проверок

4.5.5 Комментарий

Необязательное поле:

  • Свободный текст
  • Пояснение назначения платежа

Примеры:

  • "Аванс 50%"
  • "Финальный платёж"
  • "Доплата за дополнительные работы"
  • "Оплата по акту выполненных работ №5"

4.6 После добавления платежа

Что происходит:

  1. Платёж сохраняется в базу данных
  2. Автоматически записывается ФИО бухгалтера (из учётной записи)
  3. Пересчитываются проценты оплаты по этапу
  4. Пересчитывается общая сумма оплаты по объекту
  5. Платёж появляется в истории платежей этапа
  6. Возврат к детальной информации объекта

Уведомление:

Платёж добавлен
Этап: Фундаментные работы
Сумма: 900 000 ₽
Дата: 20.01.2025

ВАЖНО:

  • Платёж НЕ редактируется после добавления
  • Платёж НЕ удаляется после добавления
  • Если ошибка - нужно создать корректирующий платёж (с минусом или плюсом)

Почему нельзя редактировать:

  • Финансовая дисциплина
  • Аудит и прозрачность
  • Все операции должны оставаться в истории
  • Если ошибка - создаётся обратная операция или корректирующая запись

4.7 Что бухгалтер НЕ видит

Важные ограничения:

НЕ видит план-график работ:

  • Даты начала/окончания этапов
  • Запланированные ресурсы (люди, техника)
  • Календарный план (Gantt)

НЕ видит фактическое выполнение:

  • Ежедневные проверки ресурсов (проверяющий)
  • Проверки объёмов работ (технадзор)
  • Процент физического выполнения работ

НЕ видит аналитику план/факт:

  • Графики прогресса
  • Отклонения по ресурсам
  • Статистику выполнения

НЕ видит обращения подрядчика:

  • Проблемы на объекте
  • Переписку с министром/технадзором

Что видит:

  • Список своих объектов
  • Названия этапов работ
  • Бюджеты этапов и объектов
  • Историю платежей
  • Процент оплаты по каждому этапу
  • Общую финансовую сводку

Почему такие ограничения:

  • Разделение обязанностей (финансы отдельно от контроля выполнения)
  • Упрощение интерфейса
  • Фокус только на финансовом учёте
  • Безопасность данных

4.9 Особые случаи

4.9.1 Переплата по этапу

Ситуация:

  • Бюджет этапа: 1 800 000 ₽
  • Уже оплачено: 1 800 000 ₽ (100%)
  • Бухгалтер хочет добавить ещё платёж: 200 000 ₽

Поведение системы:

  • Показать предупреждение: "Сумма превысит бюджет этапа на 200 000 ₽"
  • Разрешить добавить платёж (не блокировать)
  • После добавления показать: "Оплачено: 2 000 000 ₽ (111%)"

Причина переплаты:

  • Дополнительные работы
  • Изменение объёма работ
  • Ошибка в планировании

4.9.2 Несколько бухгалтеров на одном объекте

Ситуация:

  • На объекте работают 2 бухгалтера (например, главный + помощник)
  • Оба привязаны к одному объекту
  • Оба могут добавлять платежи

Поведение системы:

  • Каждый видит все платежи по объекту (не только свои)
  • В истории платежей указано ФИО того, кто внёс запись
  • Не создаётся конфликтов (платежи не редактируются)

4.9.3 Корректировка ошибочного платежа

Ситуация:

  • Бухгалтер ошибся и внёс 90 000 ₽ вместо 900 000 ₽

Решение:

  • Нельзя удалить/изменить платёж
  • Нужно создать корректирующий платёж на разницу
  • Добавить платёж: 810 000 ₽ с комментарием "Корректировка платежа от 20.01.2025"

Итог:

  • В истории два платежа: 90 000 ₽ и 810 000 ₽
  • Общая сумма: 900 000 ₽ (правильная)
  • История полная и прозрачная

ЭТАП 5: КОНТРОЛЬ ОБЪЁМОВ РАБОТ (ТЕХНАДЗОР)

5.1 Описание роли "Технадзор"

Кто это:

  • Сотрудник подведомственной организации министерства
  • Технический надзор за ходом строительства
  • Контролирует фактическое выполнение работ

Основные функции:

  1. Внесение плана-графика в систему (см. ЭТАП 2)
  2. Проверка объёмов работ - фиксация процента выполнения этапов
  3. Ответы на обращения подрядчиков (первичные)
  4. Контроль прогресса по всем объектам

Это НЕ:

  • НЕ проверяет количество людей/техники (это делает Проверяющий)
  • НЕ принимает решения о платежах (это Бухгалтер)
  • НЕ утверждает планы (это Министр)

Суть роли:

  • Проверка физического выполнения работ
  • Фиксация процента выполнения каждого этапа
  • Контроль соответствия факта плану
Важная особенность:
  • Технадзор видит ВСЕ объекты республики (не привязан к конкретным)
  • НЕ видит финансы (платежи, бюджеты по этапам - видит только общий бюджет объекта)
  • Видит ежедневные проверки ресурсов (люди, техника от Проверяющего)
  • Видит план-график работ (Gantt)

5.2 Процесс работы технадзора

Как это происходит в реальности:

1. Выезд на объект

  • Технадзор приходит на стройку
  • Осматривает выполненные работы
  • Делает замеры, записи (в блокнот/телефон)
  • Фотографирует (опционально)

2. Работа с документами

  • Подрядчик предоставляет документы (акты, замеры, накладные)
  • Технадзор возвращается в офис
  • Проверяет документы
  • Считает/проверяет объёмы

3. Внесение данных в систему

  • После проверки документов
  • Вносит процент выполнения этапа
  • Добавляет комментарий (что сделано)
Частота проверок:
  • НЕ строго раз в неделю
  • В любой момент когда нужно
  • По мере выполнения работ
  • Может быть несколько проверок в неделю или одна в две недели

5.5 Форма добавления проверки объёмов

Поле Тип Обязательность Описание
Этап работ Выбор из списка Обязательно Выбор этапа из плана-графика
Дата проверки Дата Обязательно Дата фактической проверки
Процент выполнения Число (0-100) Обязательно Процент выполнения этапа
Комментарий Текст Необязательно Что сделано, замечания

5.6 Логика работы формы

5.6.1 Выбор этапа работ

Откуда берется список этапов:

  • Система показывает все этапы из плана-графика этого объекта
  • Этапы отображаются в хронологическом порядке
  • Можно выбрать любой этап (не обязательно текущий по датам)

После выбора этапа автоматически показывается:

  • Период этапа (дата начала - дата окончания)
  • Текущее выполнение (процент из последней проверки)
  • По плану на сегодня (рассчитанный процент исходя из прошедшего времени)
Пример:
Этап: Фундаментные работы
Период: 01.01.2025 - 15.01.2025 (15 дней)
Сегодня: 08.01.2025 (прошло 7 дней)
По плану должно быть: ~47% (7/15)
Последняя проверка: 45% (от 05.01.2025)

5.6.2 Дата проверки

Обязательное поле:

  • Дата фактической проверки объекта
  • Можно выбрать любую дату (прошлую, текущую, будущую)
  • По умолчанию = сегодняшняя дата

Примечание:

  • Если технадзор был на объекте вчера, а вносит данные сегодня - выбирает вчерашнюю дату

5.6.3 Процент выполнения

Обязательное поле:

  • Число от 0 до 100
  • Целое число (без десятичных)
  • Технадзор сам определяет процент на основе осмотра и документов
Валидации:

1. Процент больше 100%:

Ошибка!
Процент не может быть больше 100%.

2. Процент меньше предыдущего:

Внимание!
Вы указали 65%, а предыдущая проверка была 70%.
Процент выполнения уменьшился. Вы уверены?
[Да, продолжить] [Отмена]

3. Процент равен предыдущему:

Информация
Процент не изменился (70%).

5.6.4 Комментарий

Необязательное поле:

  • Свободный текст
  • Описание что сделано, какие работы выполнены
  • Замечания, проблемы

Примеры комментариев:

  • "Залито 128 м³ бетона из 150 м³, опалубка установлена"
  • "Работа идёт с опережением графика"
  • "Задержка из-за дождей, ожидаем нормализации погоды"
  • "Выполнены работы согласно проекту"
Рекомендация: Желательно описывать ЧТО именно сделано (в физических единицах: м³, м², шт.). Это помогает министру понимать реальную картину.

5.7 После добавления проверки

Что происходит:

  1. Проверка сохраняется в базу данных
  2. Автоматически записывается ФИО технадзора (из учётной записи)
  3. Пересчитывается общий прогресс объекта (средний процент по всем этапам с учётом их длительности)
  4. Обновляется статус отклонения (опережаем/отстаём/в графике)
  5. Проверка появляется в истории проверок этапа
  6. Возврат к детальной информации объекта

Уведомление:

Выполнение: 85%
Дата: 20.01.2025

5.8 Редактирование проверки

Логика:

  • Технадзор может редактировать любую свою проверку
  • Кнопка [Редактировать] рядом с каждой проверкой в истории
  • Открывается та же форма с заполненными данными
  • Можно изменить: дату, процент, комментарий
  • Нельзя изменить: этап (для этого нужно удалить и создать заново)

Ограничение:

  • Можно редактировать только свои проверки (не проверки других технадзоров)

Причина:

  • Вдруг технадзор ошибся в процентах
  • Вдруг появилась новая информация после внесения данных
  • Нужно дополнить комментарий

5.10 Что технадзор ВИДИТ (доступ к данным)

Полный доступ:

Видит все объекты республики:

  • Не привязан к конкретным объектам
  • Может открыть любой объект
  • Видит всю информацию по всем стройкам

Видит план-график работ (Gantt):

  • Календарный план
  • Этапы работ с датами
  • Запланированные ресурсы (люди, техника)
  • Может открыть визуализацию gantt_view.html

Видит ежедневные проверки ресурсов:

  • Данные от Проверяющих
  • Сколько людей и техники фактически на объекте каждый день
  • Историю проверок ресурсов

Видит обращения подрядчиков:

  • Проблемы на объектах
  • Может отвечать на обращения (первично)
  • Видит переписку

Видит аналитику и дашборды:

  • Графики прогресса
  • Сравнение план/факт
  • Отклонения по ресурсам и объёмам

Ограниченный доступ:

Видит ТОЛЬКО общий бюджет объекта:

  • Не видит бюджеты отдельных этапов
  • Это сделано намеренно, чтобы финансовый контроль был отдельно

НЕ видит платежи (финансы):

  • Когда и сколько оплачено
  • Процент финансирования
  • История платежей

Почему такое разделение:

  • Технадзор контролирует ФИЗИЧЕСКОЕ выполнение
  • Бухгалтер контролирует ФИНАНСЫ
  • Министр видит всё и сопоставляет данные

ЭТАП 6: ОБРАЩЕНИЯ ПОДРЯДЧИКА

6.1 Назначение системы обращений

Проблема:

  • Подрядчик столкнулся с проблемой на объекте (труба, погода, нет материалов)
  • Отправил официальное письмо министру (бумажное или email)
  • Ждёт ответа, но письмо может идти долго
  • Стройка стоит, а министр не знает о срочности

Решение:

  • Подрядчик создаёт обращение в системе параллельно с официальным письмом
  • Министр/технадзор сразу видят проблему
  • Могут оперативно ответить в системе
  • Готовят официальный ответ параллельно

Преимущества:

  • Срочность - министр сразу видит проблему, не ждёт пока письмо дойдёт
  • Прозрачность - все видят что подрядчик ждёт ответа X дней
  • Контроль - нельзя "забыть" ответить на обращение
  • История - вся переписка сохраняется

6.2 Описание роли "Подрядчик"

Кто это:

  • Юридическое лицо (ООО, ИП), выполняющее строительные работы
  • Привязан к конкретным объектам как исполнитель

Основные функции:

  1. Просмотр своих объектов - только те, где он подрядчик
  2. Просмотр плана-графика (read-only) - видит сроки и этапы
  3. Просмотр фактического выполнения - успевает или отстаёт
  4. Создание обращений - фиксация проблем и вопросов
  5. Просмотр ответов - переписка с министром/технадзором

Что подрядчик ВИДИТ:

  • Свои объекты (где он подрядчик)
  • План-график работ (Gantt) - read-only
  • Фактическое выполнение (успевает или отстаёт)
  • Проверки технадзора (процент выполнения по объёмам)
  • Ежедневные проверки ресурсов (сколько людей/техники фиксируют проверяющие)
  • Свои обращения и ответы на них

Что подрядчик НЕ ВИДИТ:

  • Финансы (платежи, бюджеты этапов, когда и сколько оплачено)
  • Другие объекты (конкурентов, других подрядчиков)
  • Обращения других подрядчиков

Что подрядчик НЕ МОЖЕТ делать:

  • Редактировать план-график
  • Редактировать проверки (ресурсов, объёмов)
  • Создавать объекты
  • Видеть/изменять финансовую информацию
  • Только: создавать обращения и видеть ответы

Зачем подрядчику видеть контроль:

  • Понимает успевает он или нет по плану
  • Видит замечания технадзора к качеству/объёмам
  • Может планировать свою работу
  • Прозрачность со стороны министерства

6.5 Форма создания обращения

Поле Тип Обязательность Описание
Этап работ Выбор из списка Обязательно К какому этапу относится проблема
Тип обращения Выбор из списка Обязательно Категория проблемы
Тема обращения Текст Обязательно Краткое описание (заголовок)
Описание проблемы Текст Обязательно Подробное описание + реквизиты письма
Дата остановки работ Дата Необязательно Если работы остановлены
Фото Файлы Необязательно До 5 фотографий

Доступные типы обращений:

  • Препятствие на объекте (труба, кабель, коммуникации)
  • Погодные условия (дождь, снег, мороз)
  • Отсутствие материалов (задержка поставки)
  • Проблема с техникой (поломка, недоступность)
  • Вопрос по проекту (несоответствие, неясность)
  • Согласование изменений
  • Другое

6.6 Статусы обращений

Жизненный цикл обращения:

1. Новое

  • Только создано подрядчиком
  • Никто ещё не прочитал
  • Требует внимания

2. На рассмотрении

  • Министр/технадзор прочитали
  • Готовят ответ
  • Могут запросить дополнительную информацию

3. Решено

  • Ответ получен
  • Проблема решается или решена
  • Подрядчик может возобновить работы

4. Закрыто

  • Вопрос полностью решён
  • Работы возобновлены
  • Архив

Автоматические переходы:

  • Подрядчик создал → статус "Новое"
  • Министр/технадзор открыли обращение → статус "На рассмотрении"
  • Министр/технадзор написали ответ → статус "Решено"
  • Министр нажал [Закрыть обращение] → статус "Закрыто"

6.9 Кто отвечает на обращения

Иерархия ответов:

1. Первично отвечает Технадзор

  • Видит все новые обращения
  • Может оперативно ответить
  • Может выехать на объект для проверки

2. Министр видит все обращения

  • Может ответить в любой момент
  • Может вмешаться в решение проблемы
  • Принимает финальные решения

3. Оба видят всю историю переписки

  • Технадзор и Министр видят все ответы друг друга
  • Подрядчик видит все ответы

Права:

  • Технадзор может отвечать
  • Министр может отвечать
  • Министр может закрывать обращения
  • Технадзор может менять статус на "Решено"
  • Подрядчик НЕ может отвечать (только создаёт обращения и читает ответы)

6.11 Уведомления

Для подрядчика:

  • Email/SMS при ответе на обращение
  • Push-уведомление (если есть мобильное приложение)

Для Министра:

  • Email при новом обращении (особенно если работы остановлены)
  • Дашборд показывает количество новых обращений

Для Технадзора:

  • Email при новом обращении по объектам
  • Уведомление если обращение не отвечено более 24 часов

6.12 Метрики и контроль

Система отслеживает:

  • Время ожидания ответа (с момента создания до первого ответа)
  • Среднее время решения обращений
  • Количество обращений по объектам (индикатор проблемных объектов)
  • Просроченные обращения (более 24 часов без ответа)

Для Министра отображается:

Статистика обращений за месяц:

• Всего обращений: 45
• Среднее время ответа: 4.2 часа
• Решено в срок (<24ч): 38 (84%)
• Просрочено: 7 (16%)
• Проблемных объектов (>5 обращений): 2

Пример использования

Сценарий: Подрядчик наткнулся на трубу

День 1, 10:00 - Проблема на объекте

  • Подрядчик копает котлован, наткнулся на водопроводную трубу
  • Работы остановлены
  • Составляет официальное письмо министру
  • Отправляет письмо (почта России, может идти 3-5 дней)

День 1, 10:30 - Обращение в системе

  • Подрядчик заходит в систему
  • Создаёт обращение:
    • Тип: Препятствие на объекте
    • Тема: "Обнаружена водопроводная труба"
    • Описание: "При копке котлована на глубине 2м обнаружена труба. Письмо №123 от 10.01.2025 направлено министру"
    • Дата остановки: 10.01.2025
    • Прикладывает фото трубы

День 1, 10:31 - Уведомление

  • Технадзор получает email: "Новое обращение по объекту Школа №5"
  • Министр видит на дашборде: "1 новое обращение"

День 1, 11:00 - Реакция технадзора

  • Технадзор открывает обращение (статус → "На рассмотрении")
  • Звонит в водоканал, уточняет информацию
  • Пишет ответ: "Связались с водоканалом, труба на их балансе. Завтра приедут специалисты для переноса. Ориентировочно работы можно возобновить 12.01.2025"
  • Статус → "Решено"

День 1, 11:05 - Подрядчик видит ответ

  • Получает SMS: "Ответ на обращение #45"
  • Читает ответ технадзора
  • Понимает план действий

День 3, 15:00 - Завершение

  • Труба перенесена
  • Работы возобновлены
  • Министр закрывает обращение (статус → "Закрыто")

ЭТАП 7: ИНТЕРФЕЙС МИНИСТРА (ГЛАВНЫЙ АДМИНИСТРАТОР)

7.1 Описание роли "Министр"

Кто это:

  • Главный администратор республики (Министр строительства КЧР)
  • Высшее должностное лицо министерства
  • Полный контроль над всеми объектами и пользователями

Главная цель:

  • Централизованный контроль всех строительных объектов республики
  • Принятие управленческих решений на основе данных
  • Координация работы подчинённых служб

Особенности роли:

  • Видит ВСЁ по республике
  • Может всё редактировать и изменять
  • Создаёт объекты, пользователей, подрядчиков
  • Полный доступ к финансам
  • Может отвечать на обращения подрядчиков
  • Выгружает отчёты и аналитику

7.2 Главный дашборд министра

Элементы дашборда:

1. Статистические карточки

  • Всего объектов
  • В работе / Завершено
  • Общий бюджет / Освоено
  • Средний прогресс
  • Количество проблем

2. Обращения подрядчиков

  • Количество новых без ответа (красный индикатор)
  • На рассмотрении
  • Решённые за период

3. Быстрые действия

  • Создать новый объект
  • Создать пользователя
  • Перейти к отчётам

4. Графики и визуализация

  • График выполнения по месяцам
  • Распределение бюджета
  • Топ проблемных объектов

7.3 Управление объектами

Фильтры и сортировка:

1. Статус:

  • Все
  • В работе
  • Завершённые
  • Приостановленные

2. Подрядчик:

  • Все
  • Выбор конкретного подрядчика

3. Отклонение от плана:

  • Все
  • Опережают (>+5%)
  • Успевают (-5% до +5%)
  • Отстают (-10% до -5%)
  • Критично отстают (<-10%)

4. Сортировка:

  • По дате создания (новые/старые)
  • По проценту выполнения (возрастание/убывание)
  • По бюджету
  • По отклонению от плана

5. Поиск:

  • По названию объекта
  • По адресу
  • По подрядчику

7.4 Детальная информация по объекту

Министр видит ВСЁ:

  • Общую статистику выполнения
  • Финансы (бюджет, освоено, осталось)
  • Привязанные пользователи
  • Обращения подрядчиков
  • Этапы работ с финансами

Ключевая особенность:

Министр видит расхождения между объёмами и финансами:

  • Может выявлять коррупцию: оплачено 50%, а сделано 70% = хорошо
  • Или наоборот: оплачено 90%, а сделано 50% = проблема

7.9 Аналитика и отчёты

Типы отчётов:

1. Сводный отчёт по всем объектам

  • Статус каждого объекта
  • Прогресс выполнения
  • Финансирование
  • Проблемы и обращения

2. Финансовый отчёт

  • Бюджет по объектам
  • Освоение средств
  • Платежи по месяцам
  • Расхождения финансы/объёмы

3. Отчёт по обращениям

  • Количество обращений
  • Среднее время ответа
  • Топ проблем
  • Статистика по подрядчикам

4. Детальный отчёт по объекту

  • План-график
  • Фактическое выполнение
  • Проверки (ресурсы, объёмы)
  • Финансы
  • Обращения

5. Отчёт для Правительства

  • Краткая сводка по республике
  • Ключевые показатели
  • Топ достижений
  • Проблемы и риски

Формат отчётов:

1. Excel (.xlsx)

  • Подробные таблицы
  • Возможность фильтрации
  • Сводные таблицы
  • Графики

2. PDF (.pdf)

  • Презентабельный вид
  • Логотип министерства
  • Подпись министра
  • Печать

7.11 Графики и визуализация

Министр видит различные графики на дашборде:

1. График выполнения по месяцам (тренд)

  • Линейный график
  • Плановое выполнение vs фактическое
  • Прогноз на будущие месяцы

2. Распределение бюджета по объектам

  • Круговая диаграмма
  • Показывает долю каждого объекта в общем бюджете

3. Освоение средств

  • Столбчатая диаграмма
  • По месяцам: план vs факт

4. Топ проблемных объектов

  • Таблица с сортировкой
  • Объекты с наибольшим отставанием

5. Статистика обращений

  • Динамика обращений по неделям
  • Среднее время ответа
  • Топ причин обращений

7.13 Что министр НЕ может делать

Единственные ограничения:

Не может создавать другие республики:

  • Только суперадмин создаёт республики

Не может видеть данные других республик:

  • Видит только свою республику (КЧР)

Не может управлять подпиской:

  • Только суперадмин управляет оплатой/тарифами

ВСЁ остальное министр МОЖЕТ:

  • Создавать/редактировать/удалять объекты
  • Создавать/редактировать/деактивировать пользователей
  • Видеть все данные (ресурсы, объёмы, финансы)
  • Отвечать на обращения
  • Выгружать любые отчёты
  • Редактировать план-график
  • Управлять настройками

Управление пользователями

Министр может создавать пользователей всех ролей:

Создание пользователя:

  • ФИО
  • Email
  • Телефон
  • Роль (Технадзор, Проверяющий, Бухгалтер, Правительство)
  • Привязка к объектам (для Проверяющих и Бухгалтеров)

Привязка к объектам:

  • Один пользователь может быть привязан к нескольким объектам
  • На одном объекте может быть несколько проверяющих/бухгалтеров
  • Технадзор видит все объекты (привязка не нужна)

Деактивация пользователей:

  • Пользователя можно деактивировать (не удалять)
  • Деактивированный пользователь не может войти в систему
  • История его действий сохраняется

Управление подрядчиками

Создание подрядчика:

  • Полное название (ООО, ИП)
  • ИНН
  • Телефон
  • Email
  • ФИО ответственного лица
  • Создание учетной записи для доступа в систему (опционально)

Редактирование подрядчика:

  • Изменение контактов
  • Изменение ответственного лица
  • Деактивация (если больше не работает)

Просмотр статистики по подрядчику:

  • Количество объектов
  • Средний процент выполнения
  • Количество просроченных объектов
  • Количество обращений

Настройки системы

Министр может настраивать:

Уведомления:

  • Время отправки напоминаний проверяющим
  • Email-адреса для получения уведомлений
  • Настройки критичности (когда отправлять срочные уведомления)

Справочники:

  • Добавление новых видов техники
  • Добавление типов обращений
  • Настройка статусов объектов

Внешний вид:

  • Логотип министерства
  • Контактная информация
  • Настройки отчётов (шаблоны)

ЭТАП 8: ИНТЕРФЕЙС ПРАВИТЕЛЬСТВА

Описание роли

Правительство (Губернатор и его аппарат) - роль для высшего руководства республики, которому нужен доступ к данным для надзора и отчётности, но без права вмешиваться в рабочие процессы министерства.

Главный принцип:

Видят ВСЁ, но изменять НИЧЕГО не могут.

8.1 Основные возможности

Доступные функции:

  • ✅ Просмотр всех строительных объектов республики
  • ✅ Просмотр всех планов-графиков (диаграммы Ганта)
  • ✅ Просмотр всех финансовых данных (бюджеты, платежи, расхождения)
  • ✅ Просмотр всех проверок (ресурсы, объёмы работ)
  • ✅ Просмотр обращений подрядчиков
  • ✅ Просмотр дашборда со статистикой и графиками
  • ✅ Выгрузка отчётов (Excel, PDF)
  • ✅ Фильтрация и поиск по объектам

Запрещённые функции:

  • ❌ Создание объектов
  • ❌ Редактирование объектов
  • ❌ Удаление объектов
  • ❌ Создание пользователей
  • ❌ Управление пользователями
  • ❌ Создание подрядчиков
  • ❌ Внесение проверок (ресурсы, объёмы, платежи)
  • ❌ Ответы на обращения
  • ❌ Любые изменения данных

8.2 Главный экран (дашборд)

Интерфейс идентичен интерфейсу Министра, но БЕЗ кнопок действий.

Блок статистики:

  • 📊 Всего объектов
  • 🏗️ В работе / Завершено / Новые
  • 💰 Общий бюджет / Выплачено
  • ⏱️ Просрочено объектов

Графики (те же, что у Министра):

1. Выполнение по срокам (круговая диаграмма)

  • Опережают график
  • В графике
  • Отстают от графика
  • Критическое отставание

2. Финансовая дисциплина (столбчатая диаграмма по месяцам)

  • План платежей
  • Фактические платежи
  • Расхождения

3. Расхождения метрик (график линейный)

  • Ресурсы (план vs факт)
  • Объёмы работ
  • Финансы
  • Среднее расхождение

4. Активность подрядчиков (таблица топ-10)

  • ФИО подрядчика
  • Количество объектов
  • Выполнение в срок (%)
  • Средний процент отклонения

8.3 Список объектов

Фильтры (те же, что у Министра):

  • По статусу (новые, в работе, завершённые, просроченные)
  • По подрядчику
  • По проценту выполнения (0-25%, 25-50%, 50-75%, 75-100%)
  • По наличию расхождений метрик
  • По датам начала/окончания
  • По району республики

ВАЖНО: Нет кнопки "➕ Создать объект"

8.4 Страница объекта

При клике на объект открывается детальная страница (как у Министра, но read-only).

Вкладки:

1. План-график (Gantt)

  • Полная диаграмма Ганта со всеми этапами
  • График ресурсов (люди, техника)
  • Цветовая индикация отклонений
  • Нет кнопок редактирования

2. Проверки ресурсов

  • История всех ежедневных проверок
  • График план vs факт
  • Комментарии проверяющих
  • Нет формы для добавления проверки

3. Контроль объёмов

  • Все проверки технадзора
  • Процент выполнения по этапам
  • График прогресса
  • Нет формы для добавления проверки

4. Финансы

  • Все платежи с датами
  • Разбивка по этапам
  • Процент оплаты
  • Расхождения с объёмами работ
  • Нет формы для добавления платежа

5. Обращения подрядчика

  • Все обращения со статусами
  • История ответов
  • Прикреплённые фото
  • Нет возможности ответить

6. Общая информация

  • Название, адрес, сроки
  • Подрядчик (контакты)
  • Бюджет и финансирование
  • Назначенные пользователи
  • Нет кнопок редактирования

8.5 Отчёты

Правительство может выгружать отчёты (как Министр):

Доступные отчёты:

1. Общий отчёт по республике (Excel)

  • Финансовые показатели
  • Прогресс по срокам
  • Расхождения метрик

2. Отчёт по объекту (PDF)

  • Детальная информация по одному объекту
  • План-график с процентами
  • Все проверки и платежи
  • Фотоотчёты

3. Финансовый отчёт (Excel)

  • Бюджеты и платежи по всем объектам
  • Расхождения с объёмами работ
  • Просроченные платежи
  • Группировка по подрядчикам

4. Отчёт по срокам (Excel)

  • Объекты в графике / с отставанием
  • Прогноз завершения
  • Критичные объекты

5. Отчёт по обращениям (PDF)

8.9 Уведомления

Правительство НЕ получает уведомления о рабочих процессах:

  • ❌ Новые обращения подрядчиков
  • ❌ Просроченные проверки
  • ❌ Отклонения метрик

Получает только сводки (опционально, настраивается):

  • ✅ Еженедельный отчёт по строительству (email)
  • ✅ Ежемесячный финансовый отчёт (email)
  • ✅ Уведомления о критических проблемах (только если Министр отметит проблему как критичную)

8.10 Зачем эта роль?

Причины существования роли Правительство:

  1. Прозрачность - высшее руководство видит реальную ситуацию
  2. Невмешательство - не мешают рабочему процессу Министерства
  3. Отчётность - могут формировать свои отчёты для вышестоящих инстанций
  4. Контроль - видят расхождения и проблемы, могут задать вопросы Министру напрямую
  5. Доверие - доступ к данным без права редактирования исключает подозрения в манипуляциях

8.11 Сценарии использования

Сценарий 1: Подготовка к совещанию

Заместитель Главы Республики готовится к совещанию по строительству:

  1. Заходит в систему
  2. Смотрит дашборд - видит общую картину
  3. Фильтрует объекты с критическим отставанием
  4. Открывает детали проблемных объектов
  5. Смотрит обращения подрядчиков и ответы Министерства
  6. Выгружает отчёт Excel для совещания
  7. На совещании задаёт Министру вопросы на основе данных

Сценарий 2: Контроль финансов

Финансовый советник Губернатора проверяет расходы:

  1. Заходит в раздел "Финансы"
  2. Смотрит общую сумму платежей за месяц
  3. Фильтрует объекты с расхождением платежей и объёмов работ
  4. Открывает детали объектов с красными индикаторами 🔴
  5. Формирует список вопросов для Министра
  6. Выгружает финансовый отчёт

Сценарий 3: Запрос от прокуратуры

Прокуратура запросила данные по конкретному объекту:

  1. Сотрудник аппарата Правительства заходит в систему
  2. Находит объект по названию
  3. Открывает детали
  4. Выгружает PDF-отчёт по объекту с планом-графиком и платежами
  5. Передаёт отчёт в прокуратуру

8.12 Безопасность

Меры безопасности для роли Правительство:

  1. Логирование всех действий - записывается каждый просмотр объекта/отчёта
  2. IP-ограничения (опционально) - доступ только из сети Правительства
  3. Двухфакторная аутентификация - обязательна для этой роли
  4. Автоматический выход - через 15 минут неактивности
  5. Запрет на API-доступ - только через веб-интерфейс, нельзя использовать API напрямую
  6. Водяные знаки на отчётах - PDF и Excel содержат ФИО пользователя и дату/время выгрузки
